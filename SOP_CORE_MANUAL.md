# Job-Filter-Wizard 核心治理手册 (Core Manual)

> **"AI 驱动逻辑，你驱动规约。"**  
> 本手册旨在为“驾驶员（用户）”提供对 Job-Filter-Wizard 2.0 体系的全局透视，重点阐述如何通过物理规约锁死 AI 幻觉，确保招聘信息抓取的准确性与去重性。

---

## 第零章：血统与哲学 —— 为什么是 Job-Filter-Wizard？

在信息爆炸的时代，我们不缺招聘信息，但缺乏**高匹配度、去重后且即时**的职位列表。

### 0.1 招聘筛选器物理全景架构 (The Three-Layer Architecture)

理解本系统的物理轨迹是高效开发的关键：

```text
┌────────────────────────────────────────────────────────────────┐
│  1. 规约层 (The Specs - 逻辑锚点)                                │
│     载体: openspec/specs/logic.md, .gemini/global_standard.md    │
│     职能: 定义筛选标准（城市、技能、时间窗口）。锁死 DeepSeek 提示词边界。 │
└───────────────┬────────────────────────────────────────────────┘
                │ (Drive)
                v
┌────────────────────────────────────────────────────────────────┐
│  2. 执行层 (The Change - 动力心脏)                              │
│     载体: openspec/changes/, /opsx: 指令集                      │
│     职能: 驱动抓取流。通过 Proposal->Design->Tasks 拓扑流执行。     │
└───────────────┬────────────────────────────────────────────────┘
                │ (Produce)
                v
┌────────────────────────────────────────────────────────────────┐
│  3. 数据层 (The Assets - 业务结晶)                              │
│     载体: storage/jobs/ (Markdown 表格), patterns/ (通用逻辑)      │
│     职能: 沉淀结构化职位数据。通过指纹算法确保增量抓取。              │
└────────────────────────────────────────────────────────────────┘
```

### 0.2 核心血统 (Lineage)

本框架是站在巨人肩膀上的作品，深度集成了以下核心引擎：

- **OpenSpec (Fission-AI)**：提供变更驱动的生命周期管理（治理底座）。
- **Superpowers (obra)**：提供原子化技能（Skills）与严苛的工程习惯（肌肉）。

> **智力底座规约**：关于上述引擎的底层物理逻辑，请参阅 [OpenSpec Core Mechanics](openspec/specs/upstream-lineage/openspec-core.md) 与 [Superpowers Skill Mechanics](openspec/specs/upstream-lineage/superpowers-core.md)。

### 0.2 为什么基于规约 (Why Protocols?)

1. **Prompt 的脆弱性**：自然语言指令容易随上下文长度增加而“腐烂”。**物理规约是刚性的**。
2. **逻辑与 UI 的分离**：通过 `logic_schema`，我们将智力从具体的、易变的视图中剥离，使其能够反哺母库。
3. **资产的可持续性**：代码会过时，但经过提纯的“Pattern（图纸）”是通用的。

### 0.3 极端纪律 (Extreme Discipline)

本框架继承了 Superpowers 最严苛的工程基因：

- **TDD 铁律**：严禁在有失败测试前编写任何业务代码。若 AI 违规先行编码，**必须物理删除该代码并重来**。禁止“适配”或“参考”违规代码。
- **调试铁律**：严禁在未定位根因前尝试修复。若 **3 次修复尝试** 均告失败，必须立即停止，转而向驾驶员发起“架构合理性”质疑。

## 第一章：逻辑刚性 —— 招聘规约如何驱动 AI

本体系通过物理规约（Physical Protocols）严格限制 AI 的行为，防止其在筛选时产生“幻觉”。

### 1.1 三大核心业务规约 (Business Protocols)

| 规约项           | 物理载体                  | 核心职能                                          | 约束效力 |
| :--------------- | :------------------------ | :------------------------------------------------ | :------- |
| **地理位置规约** | `openspec/specs/logic.md` | 强制执行精确城市过滤（如“成都”）。                | **极高** |
| **时间窗口规约** | `AGENTS.md`               | 自动计算日期，物理剔除 > 3 个月的岗位。           | **极高** |
| **指纹去重规约** | `design.md`               | 定义 `Normalize(公司)_Normalize(职位)` 为唯一键。 | **高**   |

### 1.2 物理审计锁：审计先行 (Audit-First)

AI 的每一个动作都必须在物理上“有据可查”：

- **Ops Changelog**：在修改任何代码（如修改抓取逻辑）前，必须在 `.gemini/ops_changelog.md` 记录意图。
- **Git Sentinel**：修改前强制执行 Git 状态检查。

## 第二章：数据全生命周期循环 —— 职位的抓取、去重与存储

在 Job-Filter-Wizard 中，代码是工具，**结构化的职位数据**才是核心资产。

### 2.1 职位数据的“精炼”路径 (Data Flow)

1.  **输入采集**：交互式询问用户城市、技能、行业偏好。
2.  **指纹加载 (Sentinel)**：扫描 `storage/jobs/` 下的所有 MD 文件，加载历史指纹。
3.  **智能筛选 (Search)**：构造 DeepSeek Prompt，执行多维度过滤。
4.  **增量合并**：比对指纹，仅将新职位追加至 Markdown 表格。
5.  **归档确认**：用户确认抓取结果后，执行 `/opsx:archive` 归档本次操作日志。

### 2.2 数据标准化规范 (Standardization)

为了保证数据的可读性与后续处理，所有输出文件必须遵循：

- **命名规范**：`[YYYYMMDD]_[搜索条件].md`。
- **字段规范**：包含 `岗位名称`、`公司名称`、`薪资范围`、`城市`、`行业`、`发布日期`、`来源链接`。

## 第三章：驾驶员协议 —— 人机协作的边界

### 3.1 驾驶员 (Operator) 的控制权

- **参数输入**：驾驶员负责提供核心搜索参数。
- **归档确认**：驾驶员负责对抓取结果进行最后的质量验收。

### 3.2 AI (Engine) 的执行权

- **自动化抓取**：AI 负责按照规约执行 API 调用与数据解析。
- **指纹维护**：AI 负责自动维护指纹库，确保不向驾驶员展示重复信息。

### 3.3 架构决策 (ADR)

- **ADR 驱动**：每一次非平庸的纠偏、架构选择或流程调整，都必须记录在 `openspec/decisions/`。
